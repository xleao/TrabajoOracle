## 1. ENUNCIADO DEL PROYECTO

La Práctica Calificada 4 consiste en el análisis, diseño e implementación de una solución funcional a nivel de base de datos para un caso de negocio elegido por cada grupo. La evaluación se centra en el diseño de los esquemas, la integridad de los datos, la seguridad, la programación PL/SQL mediante paquetes, las consultas operativas, los reportes para toma de decisiones y la sustentación técnica.

No se requiere desarrollar una aplicación web, móvil o de escritorio. La solución debe poder demostrarse mediante scripts SQL y PL/SQL ejecutables en Oracle 19c.

### 1.1 Objetivo general

Diseñar e implementar una solución de base de datos en Oracle 19c que permita gestionar un caso de negocio, aplicando modelado relacional, integridad de datos, seguridad, programación en el servidor, consultas operativas y reportes para la toma de decisiones.

### 1.2 Alcance mínimo

* Cada grupo trabaja sobre el caso elegido previamente.
* La solución debe implementar 25 requerimientos funcionales a nivel de base de datos.
* Los requerimientos deben estar distribuidos en 5 bloques de 5 requerimientos cada uno.
* Cada bloque de requerimientos debe implementarse en un paquete PL/SQL independiente.
* La solución debe organizarse como mínimo en 2 esquemas de Oracle.
* El informe completo debe incluir una matriz de trazabilidad que conecte requerimientos, tablas, paquetes, procedimientos, funciones, pruebas y evidencias.

### 1.3 Distribución de requerimientos

| Bloque | Cantidad | Código sugerido | Paquete obligatorio |
| --- | --- | --- | --- |
| Requerimientos que agregan valor | 5 | VAL01 a VAL05 | PKG_VALOR |
| Requerimientos de seguridad | 5 | SEG01 a SEG05 | PKG_SEGURIDAD |
| Requerimientos administrativos | 5 | ADM01 a ADM05 | PKG_ADMINISTRACION |
| Requerimientos de consultas operativas | 5 | CON01 a CON05 | PKG_CONSULTAS |
| Requerimientos de reportes para toma de decisiones | 5 | REP01 a REP05 | PKG_REPORTES |

### 1.4 Descripción de los bloques de requerimientos

**Requerimientos que agregan valor:**
Representan las operaciones principales del caso. Deben resolver procesos relevantes del negocio, como registrar operaciones, aprobar solicitudes, calcular importes, cambiar estados, validar reglas de negocio o generar transacciones importantes.

**Requerimientos de seguridad:**
Incluyen control de acceso, usuarios, perfiles, roles, privilegios, auditoría de operaciones, validación de credenciales o restricciones según responsabilidad del usuario.

**Requerimientos administrativos:**
Incluyen mantenimiento de datos maestros, catálogos, parámetros, estados, tipos, periodos, tarifas u otros elementos de configuración necesarios para operar el sistema.

**Requerimientos de consultas operativas:**
Permiten consultar información necesaria para el trabajo diario: busqueda por criterios, seguimiento de estados, disponibilidad, saldos, movimientos, historial o detalle de operaciones.

**Requerimientos de reportes para toma de decisiones:**
Generan información consolidada para supervisores, administradores o gerentes. Deben incluir agregaciones, indicadores, rankings, comparaciones por período, tendencias o análisis por categoría.

### 1.5 Organización de esquemas

La solución debe implementarse como mínimo en 2 esquemas de Oracle. Esta separación permite evaluar seguridad real, asignación de privilegios, responsabilidad sobre objetos, uso de paquetes y principio de mínimo privilegio.

Organización mínima recomendada:
1. Esquema de negocio: contiene tablas principales, tablas maestras, tablas transaccionales, secuencias, índices, vistas operativas y los paquetes relacionados con valor, administración, consultas y reportes.
2. Esquema de seguridad: contiene tablas de usuarios, perfiles, roles funcionales, bitácora, auditoría y el paquete de seguridad.

Organización opcional avanzada: los grupos pueden implementar 3 esquemas, separando negocio, seguridad y reportes. Esta opción debe justificarse en el informe y será valorada si mejora la claridad, seguridad y mantenibilidad de la solución.

### 1.6 Objetos mínimos de base de datos

* Tablas normalizadas de acuerdo con el caso elegido.
* Claves primarias, claves foráneas y restricciones de integridad.
* Constraints NOT NULL, UNIQUE y CHECK cuando correspondan.
* Secuencias para generación de claves, cuando sean necesarias.
* Índices justificados para consultas relevantes.
* Vistas operativas o gerenciales, cuando simplifiquen consultas o reportes.
* Paquetes PL/SQL con especificación y cuerpo.
* Procedimientos y funciones con parámetros, validaciones y manejo de excepciones.
* Datos de prueba suficientes para demostrar los 25 requerimientos.
* Scripts de prueba que permitan verificar el funcionamiento de la solución.

### 1.7 Restricciones importantes

* No se aceptan proyectos que solo presenten tablas y consultas simples.
* No se aceptan paquetes vacíos o paquetes que no correspondan a los requerimientos definidos.
* No se aceptan procedimientos aislados fuera de los paquetes.
* No se acepta código que no pueda ejecutarse en Oracle 19c.
* No se acepta el uso de triggers para reemplazar indebidamente la lógica principal de negocio.
* No se aceptan requerimientos ambiguos, duplicados o no demostrables.

## 2. ENTREGABLES

### 2.1 Informe completo digital

Documento principal del proyecto. Debe contener el análisis, diseño, implementación y evidencias de prueba. La matriz de trazabilidad forma parte obligatoria de este informe.

### 2.2 Informe resumen impreso

documento ejecutivo de 3 páginas incluido caratula. Debe sintetizar el caso, el modelo de datos, la organización de esquemas, los paquetes implementados, las principales consultas, los reportes y las conclusiones.

### 2.3 Script de base de datos

Archivo o conjunto de archivos SQL ordenados que permitan crear y probar toda la solución. Debe ejecutarse sin errores en Oracle 19c.

Orden sugerido del script:
1. Eliminación controlada de objetos, si corresponde.
2. Creación de esquemas o usuarios.
3. Creación de tablas.
4. Creación de constraints.
5. Creación de secuencias.
6. Creación de índices.
7. Inserción de datos de prueba.
8. Creación de vistas.
9. Creación de paquetes: especificación y cuerpo.
10. Asignación de privilegios y roles.
11. Ejecución de pruebas por requerimiento.

### 2.4 PPT de exposición

Presentación breve y técnica para sustentar el proyecto. Debe mostrar el caso, el problema, el modelo de datos, la organización de esquemas, los paquetes, las evidencias principales, los reportes y las conclusiones.

### 3.2 Recomendaciones para la redacción del informe

* Usar lenguaje claro, formal y técnico.
* Evitar requerimientos ambiguos o demasiado generales.
* Explicar las decisiones de diseño, especialmente la separación en esquemas.
* Incluir evidencias legibles de ejecución.
* Presentar el modelo de datos con suficiente claridad para que pueda ser evaluado sin revisar primero el script.
* Mantener correspondencia estricta entre requerimientos, paquetes y pruebas.
